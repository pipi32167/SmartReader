# SmartReader — Agent Guide

SmartReader is a Chrome extension (Manifest V3) that summarizes web pages and lets users ask questions about them via an AI API. It extracts page content as Markdown, sends it to an OpenAI-compatible endpoint using customizable prompts, and streams the response into a side panel.

---

## Technology Stack

- **TypeScript** — ES2022, ES modules (`"type": "module"`)
- **Vite** — Build tool and bundler
- **Vitest + jsdom** — Unit testing
- **Chrome Extension Manifest V3**
- **sql.js (WASM)** — SQLite database running inside an offscreen document
- **marked** — Markdown rendering in the side panel
- **OPFS** — Origin Private File System for persisting the SQLite database

## Architecture Overview

### Why an Offscreen Document?
Manifest V3 service workers cannot execute WASM or access OPFS. The extension uses an **offscreen document** (`src/offscreen/offscreen.ts`) solely to host `sql.js` and read/write the SQLite database to OPFS. The service worker proxies all DB operations to this document via `chrome.runtime.sendMessage`.

### Communication Flow
1. **Popup** sends `EXECUTE_PROMPT` or `SHOW_PAGE_MARKDOWN` to the **service worker**.
2. **Service worker** injects/communicates with the **content script** to get page content.
3. **Service worker** replaces prompt variables (`${html}`, `${text}`) and streams the request to the AI API.
4. **Service worker** sends `STREAM_CHUNK` / `STREAM_COMPLETE` / `STREAM_ERROR` messages to the **side panel** (filtered by `windowId`).
5. **Side panel** renders chunks with `marked` and shows copy/abort controls.

### Database Schema
- `api_config(id, base_url, api_key, model, updated_at)` — single-row config table
- `prompts(id, title, prompt, sort_order, created_at, updated_at)` — user-defined prompt templates

## Code Style Guidelines

- **Language**: UI strings, user-facing messages, and most comments are in **Chinese (zh-CN)**. Log messages often use English with a `[Component]` prefix (e.g., `[Service Worker]`, `[SidePanel]`).
- **Imports**: Use explicit type imports (`import type { … }`) where possible.
- **Strict TypeScript**: `strict: true` is enabled. Avoid `any` unless necessary.
- **Module format**: ES modules only (`"type": "module"`).
- **DOM ready**: UI entry points wait for `DOMContentLoaded` before initializing.
- **Message handlers**: Always return `true` from `chrome.runtime.onMessage` listeners when the response is sent asynchronously.

## TDD Principles

本项目的共享逻辑模块（`src/shared/` 下的纯函数与工具类）遵循 **测试驱动开发（TDD）** 原则。

### 核心流程

1. **Red**：先写测试——在实现功能之前，先编写一个会失败的测试用例，明确输入、输出和边界条件。
2. **Green**：再写实现——编写最小化的代码让测试通过，不追求过度设计。
3. **Refactor**：最后重构——在测试绿灯的前提下优化代码结构，消除重复，提升可读性。

### 实践规范

- **测试与实现同位**：每个 `*.ts` 文件旁应放置对应的 `*.test.ts`，例如 `utils.ts` ↔ `utils.test.ts`。
- **测试即文档**：用 `describe` 和 `it` 的嵌套结构清晰描述功能行为，让测试用例本身成为可执行的规格说明。
- **边界优先**：每个测试套件必须覆盖正常路径、空值/异常输入、以及边界条件（如空字符串、超大输入、特殊字符）。
- **快速反馈**：共享逻辑应保持无副作用、无外部依赖（不直接调用 Chrome API、DOM 操作或网络请求），以便在 `jsdom` 环境中快速运行单元测试。
- **Chrome API 与集成逻辑**：涉及 `chrome.*` API、Service Worker 消息流、Offscreen Document 的代码，以集成测试或手动验证为主，不强求单元测试覆盖率。

### 运行规范

- 提交前执行 `npm test`，确保所有测试通过。
- 新功能 PR 必须包含对应的测试文件（纯逻辑部分）。

## Key Conventions

### Message Types
All cross-component messages are typed in `src/shared/types.ts` via the `MessageType` object. Always add new message types there first.

### Prompt Variables
Prompt templates support two variables:
- `${html}` — Page content converted to Markdown (truncated to `MAX_CONTENT_LENGTH`, currently 30,000 characters)
- `${text}` — User’s current text selection

### Side Panel Window Filtering
The side panel only processes stream messages whose `windowId` matches the current window. This prevents cross-window interference when multiple Chrome windows are open.

### Offscreen Lifecycle
The service worker pings the offscreen document before use. If it is stale (from a previous extension load), it is closed and recreated. Initialization has a 30-second timeout.

### Service Worker Keep-Alive
During AI streaming, the service worker starts a 4-second interval that touches `chrome.storage.local` to prevent Chrome from terminating the worker mid-stream.

## Security Considerations

- **CSP**: The manifest allows `'self'` and `'wasm-unsafe-eval'` for script sources because `sql.js` relies on WASM.
- **API Key Storage**: The API key is stored locally in SQLite (OPFS). It is **not encrypted**; it is only as secure as the browser’s local storage boundary.
- **XSS Prevention**: `escapeHtml` (using a DOM text node) is used before inserting untrusted text into the UI. Markdown output from the AI is rendered with `marked`; ensure the renderer path remains sanitized if new plugins are added.
- **Content Script Injection**: The service worker injects `content-script.js` dynamically via `chrome.scripting.executeScript` when needed, rather than relying solely on `matches` in the manifest.

## Adding New Features

1. **New message flow?** Add the type to `src/shared/types.ts`, handle it in the service worker, and implement the consumer (popup / sidepanel / options / offscreen).
2. **New DB table?** Add the `CREATE TABLE` statement to `initializeSchema()` in `src/offscreen/offscreen.ts`.
3. **New UI page?** Add an HTML entry under `src/`, register it in `vite.config.ts` `rollupOptions.input`, and reference it in `manifest.json` if needed.
4. **New shared utility?** Place it in `src/shared/` and add a `*.test.ts` if it is pure logic.
